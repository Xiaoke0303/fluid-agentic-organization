# Memory Working Note

> FAO 记忆线精修坐标系
> 状态：最小闭环已勉强形成，远未精修完成
> 作用：为后续精修提供判断坐标，不是记忆系统设计方案

---

## 一句话总判断

记忆在 FAO 中不是"存储更多"，而是"在正确的时间尺度上保留正确的内容，并让它在需要追问时仍然可见"。当前最小闭环只是让成本触发的写回动作有处可去，距离"正确保留"和"可见追问"都还有明显缺口。

---

## 当前最小闭环解决了什么

- **写回动作有源有宿**：`CORRECTION-WRITEBACK.md` 定义了三种写回变体，`MEMORY-INDEX.md` 定义了接收格式，`CONTEXT-BUDGET.md` 提供了触发条件
- **成本约束与记忆系统不再是两张皮**：超支时可以降级为摘要写回或延后写回，不再是"要么全写要么不写"
- **记忆从方向标注进入最小协议层**：不是"应该保留什么"的空洞原则，而是"完整写回/摘要写回/延后写回"各自携带什么字段、进入什么位置的具体口径

---

## 当前最小闭环没有解决什么

- **没有检索入口**：`OPERATING-RULES.md` 和 `PRE-FLIGHT-SEQUENCE.md` 都声明了"可检索 MEMORY-INDEX"，但没有定义检索条件、检索范围、检索后如何影响当前判断
- **没有生命周期规则**：什么从任务态进入索引、什么从索引中废弃、什么该被遗忘，当前只有"一次性事实不进入主干"的原则，没有操作标准
- **没有审计独立通道**：纠错记录进入 changelog 是必经的，但 changelog 与 MEMORY-INDEX 的关系是什么——全文在 changelog、索引在 MEMORY-INDEX，还是两者各有保留？口径未清
- **没有短时工作记忆定义**：当前只有"长期连续性薄索引"，session 内的推理上下文、子代理返回、工具调用结果如何被临时保留和收缩，没有定义

---

## 空间维度：Session 内 / 外挂

### Session 内临时上下文

Session 内是推理发生的现场，包含：
- 中间工作上下文（临时推理产物、过程性中间结果）
- 子代理返回的原始输出
- 工具调用的原始结果
- 用户的即时反馈和修正

**判断价值**：高。支撑当前决策。
**审计价值**：低。中间推理步骤通常不需要被长期审计，除非出现争议需要回溯。
**处理原则**：session 内不保留全文，session 收口时收缩为摘要。只提取影响后续判断的高价值约束进入外挂索引，其余留在 session 日志中。

### Session 外工作记忆

进入 `MEMORY-INDEX.md` 和 `judgment-cards/` 的内容，目的是让未来任务"更聪明"——避免重复犯错、复用已验证的约束。

**什么值得外挂**：
- 跨任务复用的角色约束
- 经验证的消歧规则
- 高频判断约束
- 纠错后形成的写回结果（经分型、修正、优先级设定）

**什么虽然外挂但不应常驻可见**：
- 已废弃的旧约束（保留在"最近废弃"表中，但不进入检索主干）
- 延后写回的条目（标记为延后状态，不默认出现在检索结果中）
- 低优先级的单一事件记录

**处理原则**：薄索引。只保留影响未来判断的高价值内容，保持主干清晰。

### Session 外审计痕迹

进入 `changelog / correction log` 的内容，目的不是让未来任务更聪明，而是让未来追问有迹可循。

**什么必须进入审计通道**：
- 所有纠错记录（必经记录点）
- 责任锚定事件
- 边界暴露记录
- 状态变更轨迹

**处理原则**：全文保留。审计痕迹不追求薄，追求完整。MEMORY-INDEX 可以指向审计痕迹，但不应替代审计痕迹。

## 时间维度：初步光谱

### 瞬时态（推理过程中）

**范围**：当前中间工作上下文、子代理返回、工具调用结果、即时计算缓存。

**判断价值**：支撑当前决策，但随推理完成即失效。
**审计价值**：通常不需要保留。除非出现明显错误需要回溯推理链。

**什么该写**：不写入长期索引。session 内保留即可。
**什么该收**：推理收口时，只提取约束性结论（如"某工具不可靠"），不保留临时推理产物。
**什么该只留索引**：无。瞬时态内容不值得建立索引。
**什么该转为审计痕迹**：只有当推理链中出现责任争议时，才将相关段落转入审计日志。

### 任务态（单次任务闭环）

**范围**：从单一任务启动到收口的完整过程。任务态围绕单一任务闭环展开，不涉及跨任务的连续协调。

**判断价值**：同类任务复用。识别重复出现的约束和陷阱。
**审计价值**：中等。如果任务结果后续出现争议，需要回溯过程。

**什么该写**：任务日志（session 日志）中保留完整过程。精选后进入 MEMORY-INDEX。
**什么该收**：任务收口时，收缩为"核心约束摘要"——不是任务报告，而是对未来判断有影响的约束。
**什么该只留索引**：任务日志全文不进入 MEMORY-INDEX 主干，主干只保留索引级信息（约束项、来源、验证状态）。
**什么该转为审计痕迹**：涉及责任锚定、边界暴露、验证失败的任务记录，必须进入 correction log。

### 会话态（一次连续协作）

**范围**：一次连续的用户交互、多轮对话、可能的多次任务启动与收口。会话态服务于同一 session 内多任务之间的连续协调。

**判断价值**：跨任务偏好识别。用户在不同任务中的重复关切、反复出现的边界条件。
**审计价值**：中等。当用户质疑"你上次不是这么说的"时需要回溯。

**什么该写**：会话摘要。不是逐轮记录，而是提取跨任务的稳定约束。
**什么该收**：会话结束时，将高价值约束择记入 MEMORY-INDEX，低价值内容留在会话日志。
**什么该只留索引**：会话全文不进入主干。主干只保留"用户偏好 X 已验证"这类索引。
**什么该转为审计痕迹**：会话中的纠正、升级、边界争议记录进入 correction log。

### 跨会话态（多次会话间的可复用约束）

**范围**：经多轮验证、跨任务稳定出现的约束、规则、边界提醒。

**判断价值**：高。这是 MEMORY-INDEX 主干的真正内容——稳定角色约束、经验证的消歧规则、高频判断约束。
**审计价值**：高。这些约束如果被后续判断引用，其来源和验证状态必须可被追溯。

**什么该写**：进入 MEMORY-INDEX 主干的完整条目（完整写回）。
**什么该收**：不收缩。跨会话约束是已经验证过的"硬内容"，值得保留全文。
**什么该只留索引**：跨会话约束本身保留全文，但指向原始验证证据和审计痕迹的引用必须保留。
**什么该转为审计痕迹**：约束的验证过程、覆盖过程、纠错过程进入 correction log。

### 沉积态 / 审计态（纠错、责任、边界事件）

**范围**：不是为"继续工作"保留的内容，而是为"以后能追问"保留的内容。

**判断价值**：低。这些内容通常不直接用于未来判断，但影响对未来判断的信任。
**审计价值**：极高。当需要追问"当时怎么决定的""谁确认了这一点""错误是怎么被修正的"时，这些是唯一的证据。

**什么该写**：全文保留在 changelog / correction log 中。MEMORY-INDEX 中可以保留指向这些痕迹的索引。
**什么该收**：不收缩。审计痕迹需要完整。
**什么该只留索引**：MEMORY-INDEX 主干中只保留"某约束来源于某纠错"的索引，不保留纠错全文。
**什么该转为审计痕迹**：本身就是审计痕迹。

## 工程记忆与组织记忆的区别与张力

**工程记忆**——为了继续工作：
- 目标是让下一次任务更高效、更少重复犯错
- 形态：薄索引、高频检索、高信号噪音比
- 例子："用户偏好不展开外部案例""某工具在边界条件 X 下不可靠"

**组织记忆**——为了追问和审计：
- 目标是让责任归属、状态变更、纠错过程在需要时可追溯
- 形态：完整记录、低频检索、但检索时必须完整
- 例子："2026-04-10 某任务因成本约束降级为摘要写回""某边界争议由人类复核后升级"

**张力**：
- 工程记忆追求"薄"，组织记忆追求"全"
- 同一条纠错结果，工程侧只关心"以后不再这样"，组织侧关心"当时为什么这样、谁纠正的、纠正后验证了吗"
- 当前框架用 `MEMORY-INDEX.md` 承接工程记忆，用 `changelog / correction log` 承接组织记忆，但两者之间的引用关系（索引指向审计痕迹）尚未精修

## 记忆与成本的关系

**成本约束如何影响写回**：
- 当单次任务的成本（推理资源、工具调用、审阅时间）超出预算时，系统不能无差别保留全部记忆
- 降级为"摘要写回"：只保留核心约束，不保留完整上下文
- 延后写回：当前不进入索引，但标记触发条件和来源引用，待条件变化时再处理

**为什么"能记住"不等于"值得保留"**：
- 记住需要成本：存储成本、检索成本、维护成本
- 保留需要验证：未验证的记忆进入主干会污染未来判断
- 验证也有成本：审计痕迹需要被复核、约束需要被确认
- 因此选择性遗忘或索引收缩是成本约束下的重要机制，不是退而求其次

**FAO 的立场**：
- 不追求"记住一切"
- 追求"在成本约束下，保留最有判断价值和审计价值的内容"
- 摘要写回和延后写回是成本约束下的合法形态，不是退而求其次

## 记忆与审计的关系

**为什么有些记忆首先不是为了以后更聪明，而是为了以后能追问**：

- 当用户问"你为什么做出这个判断"时，需要回溯判断的来源和验证状态
- 当组织问"这个错误是怎么被修正的"时，需要回溯纠错流程和责任主体
- 当监管问"谁对这项决策负责"时，需要回溯责任锚定的过程

这些追问不依赖"薄索引"，而依赖"完整痕迹"。

**当前缺口**：
- `CORRECTION-WRITEBACK.md` 有错误来源（纠正者、时间、相关任务），但没有与 `changelog` 的明确衔接
- `MEMORY-INDEX.md` 有"最近写入"表，但审计追问时可能需要追溯到 correction log 的全文
- 两者之间的引用链（MEMORY-INDEX 条目 → 对应 correction log 记录）尚未建立

**审计记忆的管理模式**：
审计痕迹更接近链式、追加式记录——一旦写入不应被静默覆盖，新增记录只能追加在尾部，旧记录保留可见。工作记忆则更接近"当前有效状态 + 历史可追溯"：允许标记某些条目为延后、废弃或覆盖，但变更本身也应被记录。时间维度不仅是"保留多久"，也是"如何让后续变化可追问"——后续追问需要看到的不只是当前状态，还包括这条状态是怎么从之前的状态演变而来的。

---

## 记忆与纠错的关系

**为什么 CORRECTION-WRITEBACK 不能等于完整记忆系统**：

- CORRECTION-WRITEBACK 是纠错流程的终点之一（把纠错结果写回约束接口）
- 它不是记忆系统的起点，因为大部分纠错记录不值得进入长期索引
- 只有"经分型、修正、优先级设定后的高价值纠错结果"才进入 MEMORY-INDEX
- 完整纠错记录留在 changelog，是审计痕迹，不是工作记忆

**两者的正确关系**：
- CORRECTION-WRITEBACK 处理纠错流程（错误分型、修正、写回位置判定）
- MEMORY-INDEX 承接经处理后的高价值结果（跨任务复用的约束）
- changelog 保留完整审计痕迹
- 三者形成：纠错流程 → 结果沉淀 → 审计回溯 的链条

## 未来最值得精修的 3 个方向

### 方向 1：检索入口与激活机制

当前 `OPERATING-RULES.md` 和 `PRE-FLIGHT-SEQUENCE.md` 都声明了"可检索 MEMORY-INDEX"，但没有定义：
- 什么时机检索
- 检索什么关键词/条件
- 检索结果如何影响当前判断
- 检索不到时如何处理

这是记忆从"存进去"到"被用起来"的关键缺口。

### 方向 2：生命周期与收缩规则

当前只有"一次性事实不进入主干"的原则，没有操作标准：
- 什么从任务态进入跨会话态（验证标准是什么）
- 什么从跨会话态进入沉积态（何时标记为 legacy）
- 什么该被遗忘（选择性遗忘的触发条件）
- 延后写回的条目何时重新评估

没有生命周期规则，记忆系统会无限膨胀。

### 方向 3：审计痕迹的独立通道

当前工程记忆（MEMORY-INDEX）和审计记忆（changelog）之间的引用关系未建立：
- MEMORY-INDEX 条目能否指向对应的 correction log 记录？
- 审计追问时，能否从 MEMORY-INDEX 条目反向追溯到完整纠错过程？
- 是否需要为审计记忆建立独立的检索入口（与工程记忆不同的检索逻辑）？

审计记忆不追求高频检索，但追求"被需要时必须完整"。

## 梦扰：对记忆固定化的随机扰动

文字记录记忆，会把连续经验离散化，也会让主体被自己的记录反向塑造。

因此，记忆系统不能只有记录、索引和代谢，还需要低频的"梦扰"动作：随机抽取若干记忆片段、sidecar 线索和当前问题，不以总结为目的，而以打破固定解释、发现意外关联、松动主体自我叙述为目的。

"梦扰"不是普通复盘，不负责产出结论；也不是记忆整理，不以压缩和归档为目标。它是一种有边界的随机扰动：完全随机会变成噪音，完全确定会变成复盘。

一次"梦扰"可以低频触发，随机抽取：
- 1 条 memory-line 片段
- 1 条 cost-line 片段
- 1 条旧判断或旧任务
- 1 条外部观察
- 1 个当前问题

然后只问三件事：
1. 这些东西之间有没有一个意外关联？
2. 哪个"我以为稳定的判断"其实只是当时处境的产物？
3. 哪个记录应该被松动，而不是继续强化？

"梦扰"不直接写入 framework，不强制沉淀结论；只有在多次梦扰中反复出现的结构性观察，才允许进入 memory-line。

一句话：记录让主体连续，梦让主体不要被连续性困住。
