# GitHub协作观察：治理与路由实践

## 一、治理实践观察

**来源**：2026-03-24.md

GitHub的Pull Request流程为FAO治理原则提供了一个清晰的映射：

| 治理原则 | GitHub实践 |
|---------|-----------|
| 不伪完成 | Draft PR = 明确标记为"未完成"，不可合并 |
| 状态区分 | [WIP] / [PENDING] 标签 = 工作进行中，禁止合并 |
| 真实完成 | 仅当标记为 "ready for merge" 或 maintainer 确认时才允许合并 |

这个简单的约定解决了一个关键问题：**防止半成品被意外交付**。

当开发者创建一个Draft PR时，GitHub会禁用合并按钮——这不是技术限制，而是治理机制。它告诉所有人："这里有代码，但它还未准备好进入主干。"

## 二、协作实验观察

为验证任务化方法的可行性，设计并运行了一个GitHub协作实验，核心关注三个机制：

### 路由机制

实验采用Issue-PR双轨制：
- Issue用于任务描述与需求澄清
- PR用于方案提交与审核
- 每个任务在创建时即标注类型标签，路由规则根据标签自动分配初始处理主体

### 治理机制

治理的核心是**分歧仲裁**。当智能体主体提交的方案被标记为`needs-review`时，触发人工审核；审核意见需在PR中以结构化格式回复。如果连续两轮未能达成共识，任务自动升级至项目维护者。

### 身份模板

为避免"人格化陷阱"，为智能体主体设计了极简身份模板：
- 角色边界
- 能力边界
- 责任限制
- 交互协议

### 实验初步发现

- 明确的路由规则显著减少了无效沟通（基于观察估计，减少幅度可观）
- 锚定机制使决策追溯时间明显缩短（从需要多方询问到可快速定位记录）
- 身份模板的限制条款在实验期间被触发数次，均因边界清晰而快速解决

*注：上述数据为实验观察记录，未进行严格统计检验，仅作为方法可行性的初步证据。*

## 三、Possible Hypotheses（待验证假设）

基于上述观察，以下假设值得进一步验证：

### H1：显式状态标记可降低交付风险

假设：在智能体主体协作中，引入类似Draft PR的显式状态标记机制，可显著降低半成品被意外交付的概率。

验证方向：对比实验，比较有无状态标记机制的项目交付质量差异。

### H2：双轨制路由优于单轨制

假设：Issue-PR双轨制（需求澄清与方案提交分离）比单一提交流程更能减少返工。

验证方向：分析返工率数据，比较两种模式的效率差异。

### H3：身份模板可减少人格化误解

假设：为智能体主体配备明确的身份模板（边界声明），可减少人类协作者对其能力的误判。

验证方向：用户调研，评估身份模板对协作效率的影响。

### H4：自动升级机制可防止讨论僵化

假设：当讨论陷入僵局时的自动升级机制（如两轮未共识后升级），可防止协作陷入无限循环。

验证方向：测量讨论闭环时间，比较有无升级机制的差异。
