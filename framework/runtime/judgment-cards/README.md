# 判断卡片

> 判断性知识接口。承接"判断"元动作。

---

## 文档定位

- `judgment-cards/` 是**判断性知识接口**
- 不是长期记忆主干
- 不是治理清单
- 不是技能说明书

用于承接：这类问题通常如何判断、适用条件是什么、常见陷阱是什么、稳定表达是什么。

**关于具体领域判断卡**：`framework/judgment-cards/` 用于定义通用 judgment card 接口。具体领域判断卡应优先放入对应 domain pack（如 `domains/guarantee/judgment-cards/`），而不是长期留在 framework 核心层。

---

## 一、为什么需要 judgment cards

**`MEMORY-INDEX.md` 不适合承接大量判断性知识的原因**：
- Memory Index 是信息索引，不是判断模式仓库
- 判断性知识需要结构化字段（适用条件、陷阱、稳定表达）
- Memory Index 太薄，无法承载判断逻辑的厚度

**`TRUTH-CONTRACT.md` / `FAILURE-PROTOCOL.md` 不适合承接工作判断的原因**：
- Truth Contract 是真实性协议，规定"怎么说"，不规定"怎么判断"
- Failure Protocol 是失败报告格式，不是事前判断指南
- 两者都不回答"这类问题通常如何处理"

**需要一个位于 memory 与 assurance 之间的 sidecar 接口**：
- Memory 提供原材料
- Judgment Cards 提供处理模式
- Assurance 验证输出真实性

---

## 二、judgment card 解决什么问题

| 问题类型 | 说明 |
|----------|------|
| **这类任务通常怎样判断** | 某类问题的标准判断路径 |
| **适用条件是什么** | 什么情况下这张卡片适用 |
| **必须满足什么前提** | 使用此判断前必须确认的条件 |
| **常见陷阱是什么** | 这类判断中容易出错的地方 |
| **哪种稳定表达可以复用** | 经过验证的可复用表述 |
| **过去在哪类问题上容易翻车** | 历史失败模式提醒 |

---

## 三、judgment card 不解决什么问题

- **不替代 `MEMORY-INDEX.md`**：不存储原始信息，只存储判断模式
- **不替代 `ROLE-CONTRACT.md`**：不定义角色边界，只提供判断参考
- **不替代 `TRUTH-CONTRACT.md`**：不约束真实性，真实性由 Truth Contract 约束
- **不直接承担纠错写回协议**：写回是协议（由 `CORRECTION-WRITEBACK.md` 承接），judgment card 是可能被写回修改的承载体之一
- **不承担完整技能执行说明**：只回答"怎么判断"，不回答"怎么操作"

---

## 四、与其他文件的关系

| 文件 | 关系 |
|------|------|
| `MEMORY-INDEX.md` | Memory Index 提供历史信息，Judgment Cards 提供处理这些信息的模式。前者是原材料，后者是加工模板。 |
| `ROLE-CONTRACT.md` | Role Contract 决定"谁来判断"，Judgment Cards 提供"怎么判断"的参考。角色可以决定是否采用某张卡片。 |
| `CORRECTION-WRITEBACK.md` | Correction Writeback 是纠错写回的协议记录，Judgment Cards 是可能被纠错修改或废弃的对象。写回是动词，卡片是名词。 |
| `TRUTH-CONTRACT.md` | Truth Contract 约束所有判断的真实性声明格式，Judgment Cards 需要服从这些约束。 |
| `OPERATING-RULES.md` | Operating Rules 定义任务流程，Judgment Cards 在流程中的判断节点被调用。 |
| `TERM-MAP.md` | Term Map 解决术语歧义，Judgment Cards 使用统一术语描述判断逻辑。 |

---

## 五、judgment card 的最小字段

| 字段 | 说明 |
|------|------|
| **卡片名称** | 判断类型的简短标识 |
| **问题类型** | 这张卡片针对什么类型的问题 |
| **适用条件** | 什么情况下可以套用此判断 |
| **必要前提** | 使用此判断前必须确认的条件 |
| **判断原则** | 核心判断逻辑，3-5 条 |
| **常见陷阱** | 容易犯的错误 |
| **稳定表达** | 经过验证的可复用表述 |
| **失败提醒 / 易错点** | 历史翻车记录提醒 |
| **验证状态** | 此判断模式是否经过验证：draft / verified / deprecated |
| **来源或依据** | 经验来源，可简短 |

## 六、何时新建 judgment card

- **同类判断反复出现**：同一类判断做了 3 次以上
- **某类问题已有稳定判断方式**：团队或节点已形成共识的处理方式
- **某类错误反复发生，值得沉淀为判断接口**：翻车过 2 次以上，模式已清晰
- **某类表达已有可复用的稳定口径**：对外口径已收敛，可直接复用

---

## 七、何时不该新建 judgment card

- **只是一次性事实**：没有复用价值的单次信息
- **只是长期记忆条目**：只记录"发生了什么"，不包含"怎么判断"
- **只是失败记录但还没形成稳定判断**：只有现象，没有提炼出判断模式
- **只是技能操作步骤**：属于"怎么做"，不属于"怎么判断"

## 八、建议模板

```markdown
# [卡片名称]

## 问题类型
[这张卡片针对什么类型的问题]

## 适用条件
- [ ] 条件 1
- [ ] 条件 2

## 必要前提
- [ ] 前提 1
- [ ] 前提 2

## 判断原则
1. 原则一
2. 原则二
3. 原则三

## 常见陷阱
- 陷阱 1：...
- 陷阱 2：...

## 稳定表达
```
[可复用的标准表述]
```

## 失败提醒 / 易错点
- [日期] [场景] [错误描述]

## 验证状态
- [ ] draft（草稿）
- [x] verified（已验证）
- [ ] deprecated（已废弃）

## 来源
[经验来源，可简短]
```

---

## 九、边界声明

- **judgment cards 是判断接口，不是真理仓库**：卡片内容可被证伪、被修订、被废弃
- **judgment cards 需要服从 truth contract**：所有判断声明必须符合真实性协议
- **judgment cards 可以被 correction 改写或废弃**：纠错写回机制可以修改或淘汰过时卡片
