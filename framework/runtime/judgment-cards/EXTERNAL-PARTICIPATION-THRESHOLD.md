# 外部讨论参与阈值卡

## 1. 适用场景

- 面向 GitHub 外部观察 / issue / discussion
- 用于判断「只看 / 记录 / 参与 / 退出」
- 不用于项目内部决策

---

## 2. 触发信号

| 信号类型 | 具体表现 |
|---------|---------|
| 概念混乱 | 明确涉及 memory / identity / bootstrap / injection / runtime stability 并伴随概念误用 |
| 疑问求助 | 对方在问「为什么不稳定 / 这个概念到底是什么意思 / 应该如何理解」 |
| 澄清空间 | 存在局部澄清机会，而非纯 bug 报告或功能请求 |

---

## 3. 可参与条件（必须全部满足）

1. **能定位具体误用点**：不是笼统说「有问题」，而是能指出「这里把 X 当成 Y 用了」
2. **能用一句话说清混乱**：能用 20 字以内解释核心矛盾
3. **能做局部澄清**：不需要展开 FAO 整体理论，不需要介绍框架背景
4. **不替项目表态**：不会让人误以为我代表项目维护者发言
5. **能明确退出条件**：在发送回复前，能说出「如果对方追问 X，我就不再回复」

---

## 4. 不参与条件（任一满足即不参与）

| 条件 | 判断标准 |
|------|---------|
| 纯 bug 报告 | 只有错误现象描述，没有概念层讨论 |
| 需要讲整体理论 | 说不清楚 FAO 背景就无法澄清 |
| 涉及内部决策 | 讨论 roadmap / 优先级 / 设计决策 |
| 已入混战 | 多人争论，气氛已升级 |
| 无法区分信息来源 | 自己说不清哪些是 verified / 哪些是 guessed |

---

## 5. 参与深度

```
只观察 → 记录但不参与 → 一次性局部澄清 → 不进入多轮讨论
```

**参与时必须遵守**：
- 不提 FAO
- 不做宣传
- 不抢定义权
- 不替项目表态
- 现象语言，不用体系语言

---

## 6. 退出条件

| 触发场景 | 动作 |
|---------|------|
| 对方追问超出局部澄清范围 | 不再回复 |
| 讨论升级为争论 | 不再回复 |
| 需要进入建议层（「应该怎么做」） | 不再回复 |
| 需要说「我觉得」而非「我观察到」 | 不再回复 |
| 触碰内部决策边界 | 不再回复 |

---

## 7. 最小写回要求

**不是所有观察都写回**。只有形成明确判断后才进入 external-observation.md。

**必须包含**：
- 对方问题的本质（用我的话重新说）
- 我的判断（概念误用 / 盲区暴露 / 无关）
- 是否参与及原因
- 退出条件（如参与了）
- 暴露的盲区（最重要，没有就写「无」）

---

## 8. 最小结论

参与是选择，不是义务。不看比看错了好，不回复比回复越界好。阈值卡的作用是「卡住形式化参与」，让每一次外部发言都经过判断。
