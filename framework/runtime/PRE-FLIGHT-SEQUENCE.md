# 前置检查序列

> 承接"统筹""状态锚定"与部分"控本"元动作。任务开始前的最小检查顺序。

---

## 文档定位

- 本文是**前置检查序列**
- 不是 operating rules 的替代
- 不是 truth contract 的替代
- 不是 role contract 的替代

回答的是：节点在正式展开任务前，必须先按什么顺序完成边界、信息、判断接口与输出条件的检查。

---

## 一、为什么需要 pre-flight sequence

**role contract 只能限定职责边界，不能替代任务前置检查的原因**：
- Role Contract 回答"什么能做、什么必须升级"
- 但它不回答"进入任务前应先检查什么"
- 同一角色可能面对不同任务，需要不同的前置条件

**truth contract 只能约束表达真实性，不能替代事前分流的原因**：
- Truth Contract 约束"如何诚实表达"
- 但它不回答"是否应进入这个任务"
- 事前分流不当，再好的真实性约束也无法挽救

**judgment card 只能提供判断接口，不能替代任务开始前的条件检查的原因**：
- Judgment Card 提供"怎么判断"的模式
- 但它不回答"现在是否具备使用这个判断接口的条件"
- 没有前置检查，可能误判适用条件

**当前框架在 runtime 层仍有短板**：
- 复测显示：assurance 层已有 truth contract 支撑
- 但"进入任务前应先检查什么"仍为占位状态
- Pre-flight Sequence 是 runtime 层的最短木板

---

## 二、pre-flight sequence 解决什么问题

| 问题 | 说明 |
|------|------|
| **任务边界是否清楚** | 任务目标、范围、终止条件是否明确 |
| **角色边界是否匹配** | 当前 role contract 是否覆盖本任务 |
| **信息是否基本完备** | 是否具备进入分析阶段的最小信息集 |
| **是否已有可调用的 judgment card** | 是否存在可复用的判断接口 |
| **哪些内容只能待验证，不能直接输出** | 提前识别待验证内容，避免事后标记 |
| **是否应立即升级，而不是继续展开** | 高风险或边界不清时，前置拦截 |
| **是否已具备进入分析阶段的最小条件** | 综合判定：继续 / 升级 / 暂停 |

---

## 三、pre-flight sequence 不解决什么问题

- **不替代 role contract**：Role Contract 限定职责边界，Pre-flight 检查角色匹配度，两者层次不同
- **不替代 truth contract**：Truth Contract 约束表达真实性，Pre-flight 决定"是否进入"，两者阶段不同
- **不替代 judgment card**：Judgment Card 提供判断模式，Pre-flight 检查判断接口是否可用，两者对象不同
- **不替代 correction writeback**：Correction 处理已发生错误，Pre-flight 预防错误发生，两者时序不同
- **不替代具体任务执行本身**：Pre-flight 只检查，不执行

---

## 四、最小前置检查顺序

### 1. 任务边界检查
确认任务目标、范围、终止条件是否清楚。若边界不清，后续检查无意义。

### 2. 角色匹配检查
对照 `ROLE-CONTRACT.md`，确认当前任务是否在角色可直接承担的范围内。若角色不匹配，应升级或暂停。

### 3. 信息完备性检查
对照任务目标，检查是否具备最小信息集。若关键信息缺失，应标记 `[unverified]` 或暂停。

### 4. 既有判断接口检查
检索 `judgment-cards/` 和 `TERM-MAP.md`，确认是否存在可复用的判断接口。若有，准备调用；若无，标记需新建。

### 5. 输出条件检查
对照 `TRUTH-CONTRACT.md`，预判哪些内容可直接输出 `[verified]`，哪些只能待验证 `[unverified]` / `[inferred]`。

### 6. 升级/继续/暂停判定
综合以上检查结果，判定：
- **继续**：边界清楚、角色匹配、信息基本完备
- **升级**：边界不清、角色不匹配、高风险需人工介入
- **暂停**：信息严重不足，需补充材料后再进入

---

## 五、每一步的失败信号

| 步骤 | 失败信号 |
|------|----------|
| 1. 任务边界检查 | 任务边界不清；目标模糊 |
| 2. 角色匹配检查 | 角色越权；当前 role contract 不覆盖本任务 |
| 3. 信息完备性检查 | 关键信息缺失；最小信息集不满足 |
| 4. 既有判断接口检查 | 无可复用判断接口；术语映射缺失 |
| 5. 输出条件检查 | 输出条件不满足；待验证内容占比过高 |
| 6. 升级/继续/暂停判定 | 应升级但未升级；应暂停但强行继续 |

## 六、与其他文件的关系

| 文件 | 关系 |
|------|------|
| `ROLE-CONTRACT.md` | Pre-flight 检查角色匹配度，Role Contract 提供匹配标准。前者是检查动作，后者是检查依据。 |
| `TRUTH-CONTRACT.md` | Pre-flight 预判输出条件，Truth Contract 提供标记规则。前者是事前预判，后者是全程约束。 |
| `OPERATING-RULES.md` | Operating Rules 定义任务流程，Pre-flight 是流程开始前的检查序列。后者是前者的前置条件。 |
| `judgment-cards/README.md` | Pre-flight 检查既有判断接口是否可用，Judgment Cards 是被检查的对象之一。 |
| `CORRECTION-WRITEBACK.md` | Correction 处理已发生的判断错误，Pre-flight 通过前置检查预防错误。两者是事后与事前的关系。 |
| `ENVIRONMENT-PRECONDITIONS.md` | Pre-flight 检查任务级前置条件，Environment Preconditions 检查环境级前置条件。前者在后者之后。 |

---

## 七、在框架中的作用

**如何弥补当前测试中暴露出的 runtime 短板**：
- 复测显示：assurance 层已有 `TRUTH-CONTRACT.md` 支撑
- 但"进入任务前应先检查什么"仍为占位状态
- Pre-flight Sequence 显式定义最小检查顺序，使 runtime 层从占位变为可执行

**为什么当前阶段应优先补它，而不是先补 context budget**：
- Context Budget 是资源约束（token、工具、注意力）
- Pre-flight 是任务准入控制
- 没有准入控制，资源约束也无法挽救错误任务的展开
- 当前最短木板是"是否应进入"，而不是"进入后如何分配"

## 八、建议模板

```markdown
# Pre-flight Checklist

## 任务信息
- 任务 ID：
- 任务类型：
- 检查时间：

## 检查项

### 1. 任务边界检查
- [ ] 任务目标明确
- [ ] 任务范围清楚
- [ ] 终止条件定义

**失败信号**：任务边界不清 / 目标模糊

### 2. 角色匹配检查
- [ ] 当前 role contract 覆盖本任务
- [ ] 可直接承担的判断范围匹配
- [ ] 无不直接承担的判断被触发

**失败信号**：角色越权 / role contract 不匹配

### 3. 信息完备性检查
- [ ] 最小信息集已具备
- [ ] 关键信息无缺失
- [ ] 缺失项已标记 `[unverified]`

**失败信号**：关键信息缺失 / 最小信息集不满足

### 4. 既有判断接口检查
- [ ] 已检索 `judgment-cards/`
- [ ] 已检索 `TERM-MAP.md`
- [ ] 可复用接口已标记

**失败信号**：无可复用判断接口 / 术语映射缺失

### 5. 输出条件检查
- [ ] 预判可直接输出内容 `[verified]`
- [ ] 预判待验证内容 `[unverified]` / `[inferred]`
- [ ] 符合 truth contract 标记规则

**失败信号**：输出条件不满足 / 待验证内容占比过高

### 6. 升级/继续/暂停判定
- [ ] 综合检查结果
- [ ] 判定：继续 / 升级 / 暂停

**失败信号**：应升级但未升级 / 应暂停但强行继续

## 判定结果
- [ ] **继续**：进入任务执行
- [ ] **升级**：提交人工处理
- [ ] **暂停**：补充材料后再进入

## 备注
[补充说明]
```

## 九、边界声明

- **pre-flight sequence 约束的是任务开始前的检查顺序**：它决定"是否进入"，不决定"进入后怎么做"
- **它不直接提供专业判断内容**：它检查判断接口是否可用，不提供判断内容本身
- **它可以与 role contract、truth contract、judgment cards 共存**：各司其职，依次调用
