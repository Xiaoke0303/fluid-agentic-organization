# 长期连续性薄索引

> 承接"择记""写回"与部分"消歧"元动作。决定什么值得长期保留以影响未来判断。

---

## 文档定位

- 本文是**长期连续性薄索引**
- 不是日志仓库
- 不是 judgment card
- 不是 correction protocol

回答的是：哪些信息值得长期保留以影响未来判断，哪些信息不能直接进入主干。

---

## 一、为什么需要 memory index

**judgment card 只承接具体判断接口，不能替代长期连续性索引的原因**：
- Judgment Card 提供具体场景的判断模式
- 但它不回答"长期保留什么"
- 判断接口是频繁调用的，长期索引是稳定支撑的

**correction writeback 只处理纠错流程，不能替代长期索引主干的原因**：
- Correction Writeback 处理错误、分型、决定写回位置
- 但它不回答"写入长期主干的是什么"
- 纠错是过程，索引是结果

**日志式记忆会让节点发胖的原因**：
- 无选择地记录所有历史，导致噪音淹没信号
- 检索成本上升，有效信息被稀释
- 需要薄索引，只保留高价值内容

**当前框架仍需要一个"薄记忆主干"**：
- Judgment Card 和 Correction Writeback 已落成
- 但"什么值得进入长期索引"仍为占位状态
- Memory Index 是 Continuity 层的主干空位

---

## 二、memory index 解决什么问题

| 问题 | 说明 |
|------|------|
| **什么值得长期保留** | 跨任务可复用、影响未来判断的高价值内容 |
| **什么只能进入 incubation / 待验证区** | 未充分验证、高风险、需观察后再决定的内容 |
| **什么只应进入 changelog / correction log** | 一次性事件记录、审计轨迹，不进入长期索引 |
| **什么可以影响未来检索优先级** | 高频、稳定、验证过的判断约束和消歧规则 |
| **什么不应进入长期主干** | 一次性事实、临时状态、局部上下文、未经处理的原始记录 |

---

## 三、memory index 不解决什么问题

- **不替代 judgment card**：Judgment Card 是具体判断接口，Memory Index 是长期索引主干，两者分层不同
- **不替代 correction writeback**：Correction Writeback 处理纠错流程，Memory Index 承接经处理后的高价值结果，两者是过程与结果的关系
- **不替代 truth contract**：Truth Contract 约束表达真实性，Memory Index 约束保留什么，两者对象不同
- **不替代 role contract**：Role Contract 限定职责边界，Memory Index 提供历史信息支撑，两者功能不同
- **不替代具体任务历史日志**：Memory Index 只保留索引，不保留完整日志，日志另存

---

## 四、最小择记原则

### 1. 只保留影响未来判断的内容
不保留与未来判断无关的信息。每一个进入主干的条目都必须说明：它如何影响未来的判断或决策。

### 2. 未验证内容不得直接进入主干
所有进入 `MEMORY-INDEX.md` 的内容必须经过验证。未验证内容只能进入 incubation / 待验证区。

### 3. 具体判断模式优先写入 judgment card，而不是主干记忆
具体的场景化判断模式应写入 `judgment-cards/`，而非本文件。本文件只保留跨场景的索引级信息。

### 4. 纠错结果先经 writeback 协议，再决定是否进入主干
纠错结果必须先经过 `CORRECTION-WRITEBACK.md` 处理（分型、修正、优先级设定），然后再决定是否进入本索引。

### 5. 高频、稳定、跨任务可复用的内容优先进入索引
满足以下全部条件的内容优先进入：跨任务出现、形态稳定、经充分验证、对未来检索有决定作用。

### 6. 一次性事实、临时状态、局部上下文不进入主干
单次任务的事实、临时状态、仅适用于特定上下文的细节，不进入长期主干，仅在任务日志中保留。

## 五、写入位置分流

| 位置 | 进入条件 | 内容类型 |
|------|----------|----------|
| **`MEMORY-INDEX.md`** | 经验证、跨任务可复用、影响未来判断的高价值内容 | 长期索引主干：稳定角色约束、经验证的消歧规则、高频判断约束 |
| **`judgment-cards/`** | 具体场景的判断模式、可复用的判断逻辑 | 判断接口：具体场景的处理模式、适用条件、常见陷阱 |
| **`CORRECTION-WRITEBACK.md`** | 错误分型、修正表达、写回位置判定 | 纠错协议层：处理错误的流程和规则 |
| **incubation / 待验证区** | 未充分验证、高风险、需观察后再决定 | 待孵化内容：新观察、未经验证的假设、实验性规则 |
| **changelog / correction log** | 所有事件的审计轨迹、一次性记录 | 日志层：操作记录、错误暴露记录、时间线 |

**核心区别**：
- **长期主干**（MEMORY-INDEX）：薄、验证过、跨任务、影响检索
- **判断接口**（judgment-cards）：具体场景、模式化、直接调用
- **纠错记录**（CORRECTION-WRITEBACK）：流程性、一次性、审计用途

---

## 六、长期主干应保留的内容类型

- **稳定角色约束**：经长期验证的角色边界约定
- **经验证的高频判断约束**：多次出现且验证有效的判断限制
- **跨任务复用的边界提醒**：在不同任务中反复出现的边界注意事项
- **已确认的高价值消歧规则**：术语、概念、结构的消歧规则（可指向 TERM-MAP）
- **对未来检索有决定作用的写回结果**：经 writeback 协议处理后的高优先级纠错结果

---

## 七、不应进入长期主干的内容类型

- **一次性事实**：仅与单次任务相关的事实细节
- **临时任务状态**：某次任务的中间状态、未完成标记
- **未验证纠错**：未经充分验证的纠错记录
- **具体技能操作步骤**：具体工具的使用步骤、操作指南（应入 skills）
- **只是某一次失败的原始记录**：未经分型和处理的原始失败记录（应在 correction log）

## 八、与其他文件的关系

| 文件 | 关系 |
|------|------|
| `ROLE-CONTRACT.md` | Memory Index 提供历史约束信息，Role Contract 决定当前任务如何收窄。前者是历史支撑，后者是当前约束。 |
| `judgment-cards/README.md` | Memory Index 是长期主干，Judgment Cards 是具体判断接口。前者薄，后者厚；前者索引级，后者模式级。 |
| `CORRECTION-WRITEBACK.md` | Correction Writeback 处理纠错流程，Memory Index 承接经处理后的高价值结果。前者是过程，后者是结果存储。 |
| `TRUTH-CONTRACT.md` | Truth Contract 约束进入 Memory Index 的内容必须真实，Memory Index 为 Truth Contract 提供历史验证依据。 |
| `OPERATING-RULES.md` | Operating Rules 决定何时检索 Memory Index，后者提供检索内容。前者是查询时机，后者是查询对象。 |
| `PRE-FLIGHT-SEQUENCE.md` | Pre-flight 检查信息完备性时，可检索 Memory Index 获取历史约束信息。前者是检查动作，后者是检查依据来源之一。 |

## 九、在框架中的作用

**如何补上当前 Continuity 层的主干空位**：
- Judgment Card 和 Correction Writeback 已落成
- 但"什么值得进入长期索引"仍为占位状态
- Memory Index 显式定义择记规则，使 Continuity 层形成完整链条：纠错 → 分流 → 索引

**为什么它不是知识库，而是薄索引**：
- 知识库追求全面，薄索引追求精准
- 只保留高价值、跨任务、影响判断的内容
- 保持主干清晰，避免信息肥胖

**为什么它应放在 judgment card 与 correction writeback 之后补**：
- Judgment Card 解决"怎么判断"
- Correction Writeback 解决"错误如何处理"
- Memory Index 解决"什么值得长期保留"
- 前两者是流程，后者是结果沉淀
- 没有前两者，索引将缺乏高质量内容来源

## 十一、边界声明

- **memory index 约束的是"长期保留什么"**：它决定什么进入主干、什么分流到其他位置
- **它不直接承载具体判断正文**：具体判断正文在 judgment cards，本文件只保留索引级信息
- **它可与 judgment cards、correction writeback、truth contract 共存**：各司其职，形成完整的连续性层
