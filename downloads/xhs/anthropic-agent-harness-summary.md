# Anthropic Agent Harness 经验分享总结

> 来源：Anthropic 官方博客《Harness Design for Long-Running Application Development》
> 整理时间：2026-03-27

---

## 核心观点

**模型负责"做事"，Harness 负责让它在很长、很复杂、还容易跑偏的工作流程里，尽量一直做对事。**

---

## 什么是 Harness？

Harness（执行框架/调度框架）不是简单的"工具链"，而是一套专门为**长程任务**设计的运行机制：

- 什么时候拆任务
- 什么时候交接上下文
- 什么时候引入新的 Agent
- 生成结果后由谁来检查
- 检查不过如何打回重做
- 长上下文撑不住时是压缩历史还是直接重置会话

---

## 三层 Agent 架构

Anthropic 采用的架构：**Planner（规划者）+ Generator（生成者）+ Evaluator（评估者）**

| 角色 | 职责 |
|------|------|
| **Planner** | 拆解任务，制定执行计划 |
| **Generator** | 执行具体任务，生成结果 |
| **Evaluator** | 独立评估结果质量，提供反馈 |

**关键洞察**：把"干活的 Agent"和"评判它的 Agent"分开，能解决自我评估失真的问题。

---

## 为什么简单 Agent 容易失灵？

### 问题1：Context Anxiety（上下文焦虑）
- 上下文一长，模型就开始"慌"
- Claude Sonnet 4.5 的 context anxiety 已经强到只靠压缩根本不够
- **解决方案**：Context Reset 成为必要组件

### 问题2：Self-Evaluation 失真
- Agent 评价自己刚做出的成果时，往往会很自信地夸自己的工作
- 即便在人类看来质量只是平平
- **解决方案**：引入独立的 Evaluator，把"生成"和"评估"分离

## 关键工程经验

### 1. Context Engineering 基础设施
- 现代 AI agent 的输入输出 token 比例可达 **100:1**
- KV 缓存至关重要：使用缓存后，Claude Sonnet 输入成本可从 $3/百万 token 降到 $0.3，相差 **10倍**
- **三条铁律**：
  - 保持提示前缀稳定
  - 让上下文只追加不修改
  - 在关键位置明确标记缓存断点

### 2. Progressive Disclosure（渐进式披露）
来自 1990 年代 Nielsen Norman Group 的可用性研究：
- **第一层**：只加载技能名称和描述（元数据）
- **第二层**：匹配到用户需求时才加载完整技能内容
- **第三层**：执行过程中按需引用支持文件

### 3. 结构化产物（Structured Artifacts）
在不同会话之间完成上下文交接的关键：
- 把构建过程拆成可处理的小块
- 通过结构化产物在不同会话之间传递状态

## Harness 瘦身原则

**"模型越强，Harness 也值得做，但 Harness 里的每一个组件都隐含着一个假设——'模型自己还做不到这件事'，而这种假设需要不断做压力测试。"**

瘦身方法：
- 每次只删一个组件，看最终结果发生什么变化
- 新模型发布后，重新审视已有 Harness
- 把那些已经不再"承重"的部件剥掉

## 实战案例：4小时构建网页版 DAW

用三层架构让 Claude 在 4 小时内构建了一个网页版数字音频工作站（DAW）：
- 可以写旋律、和声、鼓点
- 把它们排成一首歌
- 同时得到一个集成 Agent 的帮助

**QA 的价值**：
- Generator 单独跑时仍然会漏细节
- 或把功能做成占位壳子就算完工
- QA 会把这些尾部缺口揪出来，再交还给 Generator 去补

---

## 四条核心经验

1. **始终亲自和模型打交道**，读取它在真实问题上的 trace，再围绕你想要的结果去调性能

2. **在复杂任务上，把任务拆开**，让不同 Agent 各自专职处理问题的不同方面

3. **新模型出现后，重新审视已有 Harness**：剥掉不再"承重"的部件，加上此前做不到的新部件

4. **先找尽可能简单的方案，只有在必要时才增加复杂度**

---

## 一句话总结

> **随着模型进步，值得探索的 Harness 组合空间并不会缩小，它只是会移动。对 AI 工程师来说，真正有意思的工作，就是不断去找到下一组新的、有效的组合方式。**
