## 什么是 Harness 2.0？

**Harness**（马具/驾驭层）是围绕 AI Agent 构建的完整基础设施系统，让模型能够在**长时间、复杂任务**中可靠工作。

**核心等式**：
> **自主权 = f(背压)**  
> 不是模型越强就能放手，是约束和反馈越完善才能放手。

---

## 三层架构（GAN 灵感）

Anthropic 借鉴生成对抗网络（GAN）思路，设计了三个智能体协同架构：

| 角色 | 职责 | 关键能力 |
|------|------|----------|
| **Planner（规划器）** | 接受 1-4 句话提示，扩展为完整产品规范 | 范围雄心、专注产品背景和技术设计 |
| **Generator（生成器）** | 以 Sprint 为单位实现功能，自评估后交 QA | React/Vite/FastAPI/SQLite 技术栈 |
| **Evaluator（评估器）** | 用 Playwright MCP 像真实用户一样测试应用 | 测试 UI、API、数据库，按标准打分 |

**Sprint 契约机制**：生成器和评估器在写代码前先协商"完成"标准，搭起用户故事和可测试实现之间的桥梁。

**提示词**：
> "用 Web Audio API 在浏览器里构建一个功能完整的 DAW（数字音频工作站）"

## 架构演进：模型变强，脚手架该减

### 从 4.5 到 4.6 的变化

| 组件 | Opus 4.5 | Opus 4.6 |
|------|----------|----------|
| Sprint 结构 | 必需（防止上下文焦虑） | 可移除（模型能连续处理） |
| 上下文重置 | 必需 | 不再需要 |
| 评估器作用 | 每轮 Sprint 后检查 | 仅在构建结束后扫描 |
| 规划器 | 保留 | 保留（去掉会低估范围） |

**关键洞察**：评估器不是固定组件——当任务落在模型能力边界内时，它是多余开销；当任务超出边界时，它带来真实提升。

## OpenAI 的三大支柱

### 支柱一：上下文工程（代码库层面）

- **AGENTS.md 瘦身**：从千页说明书缩减到 100 行目录，指向具体文档
- **运行中的应用可见**：通过 Chrome DevTools Protocol 让 Agent 获取 DOM 快照、截图
- **可观测性栈**：LogQL 查日志、PromQL 查指标，每个 worktree 独立

### 支柱二：架构约束

- **强制依赖方向**：Types → Config → Repo → Service → Runtime → UI
- **自定义 Linter**：违反架构时，错误消息包含修复指引
- **严格分层**：Providers 统一接口管理跨领域关注点

### 支柱三：Garbage Collection（代码清理）

- **AI Slop 问题**：Agent 会复制仓库里的坏模式
- **自动清理机制**：后台 Codex 任务定期扫描偏差、更新质量分数、开重构 PR
- **黄金原则编码**：
  1. 优先用共享 utility 包，别手写 helper
  2. 不要 YOLO 式探测数据结构，要么 validate boundary，要么用 typed SDK

## LangChain 的硬数据

**Terminal Bench 2.0 实验**：
- 同一个模型，只改 Harness
- 成绩从 **52.8%** → **66.5%**
- 排名从 **Top 30** → **Top 5**

**改了什么**：
1. 自验证循环（Agent 完成后自己检查）
2. 启动时映射目录结构
3. 循环检测（重复操作时强制打断）
4. Reasoning Sandwich（高推理模型规划+验证，普通模型实现）

## 开发者行动建议

### Hashimoto 的六阶段采用旅程

1. **起步阶段**：把同一个任务做两遍（手动完成 → 让 Agent 重做）
2. **养成习惯**：每天下班前 30 分钟启动 Agent（深度调研/并行探索/Issue 分诊）
3. **关键跃迁**：建一份 **AGENTS.md**
   - 核心架构说明
   - 常见 Agent 错误及应对
   - 测试和 Lint 命令
   - Agent 绝对不能碰的部分

**心态建议**：
> "关掉 Agent 的桌面通知... 作为人类，我的职责是控制何时中断 Agent，而非被它中断。"
