# FAO (Fluid Agentic Organization)

> 一个正在演化中的组织框架项目，关注在人类主体与智能体主体共存条件下，如何通过边界、路由与治理，重构协作方式。

当人类与智能体共同参与任务时，组织开始面对一类新的现实问题：

- 任务可以流动，但责任未必能流动
- 输出可以生成，但判断未必被承担
- 工具可以调用，但执行未必真实发生
- 协作可以加速，但边界可能变得模糊

FAO 试图处理的，不是"如何让 AI 多做一点"，  
而是"在异质主体协作条件下，如何重新安排任务、责任与治理"。

它关心的不只是效率，  
更关心在协作链条中：

- 什么可以被交给别的主体
- 什么不能被交出去
- 什么可以路由
- 什么必须保留责任锚点
- 当结果出错时，问题应如何被识别、暴露与承担

---

## FAO 不是在做什么

FAO 不是一个通用的 agent workflow 模板。  
它也不是一个把人类工作简单拆给多个 AI 的自动化口号。

它关注的是另一类问题：

- 在不同主体能力并不对称的情况下，任务应如何路由
- 在执行能力增强但责任关系模糊时，边界应如何保留
- 在输出越来越容易生成时，真实性与失败应如何被显化

所以，FAO 更接近一种面向人类主体与智能体主体共存条件的组织框架，  
而不是一个只服务于"提效"的编排技巧集合。

### [whitepaper/](whitepaper/)

完整问题陈述与理论展开，讨论：

- 异质主体
- 不能流动的边界
- 任务路由
- 治理与责任安排

适合想先理解问题背景和整体框架的人。

### [toolkit/minimal-core/](toolkit/minimal-core/)

最小运行单元，提供三个基础层：

- **soul**：方向层
- **memory**：连续性层
- **heartbeat**：节律层

它不是一套庞大的系统，  
而是一个尽量轻、尽量稳的最小组织骨架。

### [toolkit/governance/](toolkit/governance/)

治理层，处理那些在智能体协作中最容易被忽略、但又最容易失真的问题，例如：

- 外部调用是否真的执行
- 结论是否真的被验证
- 失败是否被完整暴露
- 责任边界是否被伪装成"已完成"

这部分不是为了增加复杂度，  
而是为了防止协作系统滑向"伪完成""伪确认""伪执行"。

### [toolkit/governance/templates/](toolkit/governance/templates/)

节点接入模板，帮助不同主体在进入协作系统时形成基本锚点，例如：

- 身份
- 用户关系
- 接入边界
- 最小上下文约束

## 为什么会有这个项目

随着智能体越来越多地进入真实工作流，  
组织面对的已经不只是"工具升级"，  
而是主体条件本身发生了变化。

过去很多默认前提正在失效，例如：

- 做事的人，未必是承担责任的人
- 产生输出的主体，未必理解输出的后果
- 调用工具的过程，未必真的发生
- 协作表面上变快了，但判断链条可能更脆弱了

FAO 的出发点，就是承认这些变化，  
并尝试在这种前提下重新理解组织。

---

## 项目当前关注的问题

目前，FAO 主要聚焦在以下几个方向：

1. **边界问题**  
   什么可以流动，什么不能流动。

2. **路由问题**  
   在不同主体之间，任务应如何分配与转接。

3. **治理问题**  
   当执行、判断、输出彼此分离时，真实性如何保障，失败如何暴露。

4. **最小结构问题**  
   一个组织如果要与智能体长期共存，最少需要什么样的稳定骨架。

## 仍然开放的问题

FAO 还远未完成。  
它目前仍处在一个持续观察、持续修正的阶段。

一些仍然开放的问题包括：

- 哪些责任可以部分转移，哪些责任不能
- 路由逻辑是否能够形成稳定模式
- 治理约束如何保持轻量，而不反过来压垮协作
- 最小组织骨架如何避免继续膨胀
- 人类主体与智能体主体之间，长期会形成怎样的新分工

---

## 说明
