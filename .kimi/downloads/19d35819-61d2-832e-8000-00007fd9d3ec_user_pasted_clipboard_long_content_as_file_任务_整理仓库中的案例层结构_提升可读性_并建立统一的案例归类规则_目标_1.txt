任务：整理仓库中的案例层结构，提升可读性，并建立统一的案例归类规则。

目标：
1. 先解决“案例放哪”的问题
2. 再解决“案例怎么写更清楚”的问题
3. 不扩展白皮书，不新增理论
4. 不把 observation / case / hypothesis / notes 混在一起

--------------------------------
一、目录与分类规则
--------------------------------

请将案例相关材料按以下四类整理：

1. observations/
用于放公开观察样本
特点：
- 来源多为新闻、访谈、公开文章
- 信息不完整
- 只能提供信号
- 不构成结构证明

2. cases/
用于放相对完整、可复盘的案例
特点：
- 过程相对完整
- 可以支持较明确的结构观察
- 允许包含“案例启发”，但不直接上升为框架结论

3. hypotheses/
用于放从 observation 或 case 中长出的待验证判断
特点：
- 不是事实
- 不是结论
- 不可直接进入白皮书主结构
- 必须显式标记为 hypothesis

4. notes/
用于放内部演化记录、development log、协作过程记录
特点：
- 不承担“对外说明结构”的任务
- 只记录我们如何走到这里

--------------------------------
二、法律案例的特殊处理
--------------------------------

对于判决书、裁判文书、司法案例：

默认归入：
cases/

但必须增加以下标签块：

Case Type: Legal / Court Judgment
Evidence Strength: High（事实层）
Bias: Post-event / extreme scenario

并遵守一条原则：

判决书不用于证明结构，
只用于校准责任边界与不可犯错误。

换句话说：
- 可以用来校准“责任最终落在哪”
- 可以用来识别“哪些错误不可原谅”
- 不能直接用一个判决书推出一般性组织结构结论

--------------------------------
三、现有材料整理任务
--------------------------------

请检查当前未合并和已存在的案例/观察相关材料，
对每份材料做三件事：

1. 判断它应归入：
- observations
- cases
- hypotheses
- notes

2. 如位置不合适，提出最小调整方案
- 移动文件
- 修改标题
- 增加说明段落
- 不重写全文，除非必要

3. 输出一个整理建议清单

格式如下：

[文件名]
- 当前定位：
- 建议定位：
- 是否需要移动：
- 是否需要增加说明：
- 是否需要改标题：
- 理由：

--------------------------------
四、可读性统一规则
--------------------------------

请为 observations / cases / hypotheses 统一增加更清晰的开头说明。

每类文件都要在开头增加一个很短的“定位说明”。

1. observations 类：
建议说明类似：
This document records a public observation sample.
It provides signals, not proof.

2. cases 类：
建议说明类似：
This document records a case that is sufficiently complete for structural observation.
It supports analysis, but does not by itself establish a general rule.

3. hypotheses 类：
建议说明类似：
This document records a hypothesis derived from observation or case material.
It is not yet a validated conclusion.

4. notes 类：
建议说明类似：
This document records internal development history.
It is not a general pattern.

要求：
- 每段 2–3 行
- 不写长解释
- 不写理论语言
- 不写“伟大意义”

--------------------------------
五、案例正文可读性修正
--------------------------------

对 observation / case / legal case 文件，统一检查以下问题：

1. 是否一上来就下结论
如果是，请改成：
- 基本信息
- 可确认事实
- 当前可见边界
- 可以提出的观察
- 尚未解决的问题

2. 是否把 hypothesis 写成了事实
如果是，请改成：
- 单独移到 hypothesis 小节
- 或移入 hypotheses/ 目录

3. 是否段落过长、读者不易进入
请做轻量整理：
- 压短段落
- 小标题更明确
- 一段只讲一件事

4. 是否出现“既像案例又像理论”
如果是，请优先保留案例，删弱理论口气

--------------------------------
六、必须遵守的约束
--------------------------------

- 不新增框架
- 不修改白皮书正文
- 不把 hypothesis 升格成结构
- 不为了整齐而重写大量内容
- 以“最小修改”优先

--------------------------------
七、输出要求
--------------------------------

先不要直接批量修改。

请先输出两部分内容：

1. 案例归类建议总表
2. 需要修改的最小动作清单

等人工确认后，再执行文件修改与 Git 操作。
