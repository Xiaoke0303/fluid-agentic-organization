# Failure Report Checklist

> 失败后的结构化报告清单。

---

## 核心检查项

- [ ] **明确承认未完成** — 不绕弯，直接说"我未完成 X"
- [ ] **明确说明缺失能力或权限** — 具体是什么工具/权限/配置缺失
- [ ] **列出已尝试步骤** — 实际做了哪些排查/尝试
- [ ] **明确指出阻塞点** — 卡在哪一步，错误信息是什么
- [ ] **区分"内容不存在"和"我无法确认"** — 不因为访问失败就说目标不存在
- [ ] **列出继续完成所需条件** — 要补齐什么才能继续

---

## 自主排查检查

- [ ] **已完成最小自主排查** — 路径/大小写/目录/同目录检查/线索/替代入口
- [ ] **未完成排查前不向人求助** — 已尽职排查后才决定是否求助

---

## 禁止事项

- [ ] **没有用模糊语气掩盖失败** — 如"似乎""可能""应该"替代明确的失败声明
- [ ] **没有把路径猜测失败说成目标不存在** — 404 ≠ 目标不存在，可能是路径错误
- [ ] **没有把"准备做"说成"已尝试"** — 意图 ≠ 行动

---

## 报告模板

```
失败报告：
- 未完成：[具体任务]
- 原因：[为什么没完成]
- 已尝试步骤：
  1. [步骤1]
  2. [步骤2]
- 阻塞点：[卡在哪一步，错误信息]
- 区分说明：[内容不存在 vs 我无法确认]
- 继续条件：[需要补充什么]
```

---

## 示例

```
失败报告：
- 未完成：获取 toolkit/minimal-core/README.md
- 原因：web_fetch 返回 404
- 已尝试步骤：
  1. 尝试标准 raw URL 路径
  2. 尝试大小写变体 README.md
  3. 尝试目录页访问（被 block）
- 阻塞点：文件不存在于预期路径
- 区分说明：无法确认是路径错误还是文件确实不存在
- 继续条件：确认文件实际路径，或创建新文件
```
